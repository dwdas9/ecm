
# IACustomWebServices - Building the SOAP Web Service and Consuming It from Captiva
## Server-Side Implementation, Proxy Generation, and Client-Side Consumption


**Prepared By:** Dwaipayan Das. Architect for the XXXXbank AFU Project.

## Preliminary Notes

This article covers two separate pieces of code:

i) **The web service** (`Service1.asmx`) - the server-side ASP.NET code that received SOAP calls from Captiva, connected to Oracle, and returned validation results.

ii) **The proxy class** (`Service1.vb`, client side) - the auto-generated VB.NET class that the Captiva IndexPlus DLLs used to call the web service. Generated by `wsdl.exe` from the live WSDL. This is not the same file as the server-side `Service1.vb`. Same filename, completely different purpose.

These two pieces together form the complete SOAP integration between Captiva and the bank's Oracle backend.

## Part 1 - The Web Service: Server-Side Implementation

### 1.1 What It Was

The web service was a standard ASP.NET ASMX web service, written in VB.NET, hosted on IIS on the Captiva server at `[SRV-MASKED]:81` under the application `IACustomWebServices`.

The class declaration:

```vb
<System.Web.Services.WebService(Namespace:="http://tempuri.org/")>
<System.Web.Services.WebServiceBinding(ConformsTo:=WsiProfiles.BasicProfile1_1)>
<ToolboxItem(False)>
Public Class Service1
    Inherits System.Web.Services.WebService
```

Two things worth noting:

- `WsiProfiles.BasicProfile1_1` - WS-I Basic Profile 1.1 conformance. This made the service interoperable with any SOAP client regardless of platform or language. The ALPS team and any other future consumer could generate a client stub from the WSDL without compatibility issues.
- `Namespace:="http://tempuri.org/"` - the default Visual Studio namespace. In a production banking service this should have been a proper URI like `http://[bank-domain]/AFU/`. `<TBD DAS - this was never changed. It worked but was not ideal.>`

The imports required:

```vb
Imports System.Web.Services
Imports System.Web.Services.Protocols
Imports System.ComponentModel
Imports System.IO
Imports ADODB
Imports System.Globalization
```

`ADODB` - the COM-based ActiveX Data Objects library - was used for Oracle connectivity. This was a reference to `msado15.dll` (`Microsoft ActiveX Data Objects 2.8 Library`). `System.IO` was needed for the `File.ReadAllLines()` calls against the INI files. `System.Globalization` was needed for the `CultureInfo("en-GB")` date parsing in `ValidateValidTill`.

### 1.2 Oracle Connection Management

The Oracle connection string was not hardcoded. It was read from `web.config` at the start of each method:

```vb
Dim conStringSettings As System.Configuration.ConnectionStringSettings _
    = System.Configuration.ConfigurationManager.ConnectionStrings("CustomerDatabase")
Dim conStr As String = conStringSettings.ConnectionString
```

The key `CustomerDatabase` mapped to the Oracle connection string in `web.config`. No Oracle credentials were visible anywhere in the service code or in any Captiva configuration screen.

Connection and command objects were handled carefully to prevent leaks. Both `con` and `cmd` were closed and set to `Nothing` in two places - in the main `Try` block after successful execution, and again in the `Catch` block:

```vb
' After successful execution:
If (Not con Is Nothing) Then
    If (Not con.State = ObjectStateEnum.adStateClosed) Then
        con.Close()
        con = Nothing
    Else
        con = Nothing
    End If
End If

' Same block repeated in Catch:
If (Not con Is Nothing) Then
    If (Not con.State = ObjectStateEnum.adStateClosed) Then
        con.Close()
        con = Nothing
    Else
        con = Nothing
    End If
End If
```

This pattern was used because ADODB connections do not have automatic disposal like .NET `IDisposable` objects. If the `Catch` block did not explicitly close the connection, any exception during execution would leave an open Oracle session. With 20+ concurrent operators triggering validation calls, leaked connections would accumulate quickly and hit the Oracle listener limit.

### 1.3 Method 1: ValidateSL210PQ - Full Source Walkthrough

```vb
<WebMethod()>
Public Function ValidateSL210PQ(ByVal strIn As String) As String
```

**Constants defined inside the method:**

```vb
Const PASS_NEWACCOUNT    As String = "1000"
Const PASS_VALIDACTNUM   As String = "1001"
Const PASS_INVALIDACTNUM As String = "1002"
Const FAIL_EXCEPTION     As String = "1003"
Const FAIL_EOFUNCREACHED As String = "1004"
Const UNIVERSAL_DELIMITER As String = "|%|%"
Const PROCEDURE_NAME     As String = "CALLSL210PQ"
Const TCODES_FILE        As String = "C:\\CaptivaCustom\\DLL\\TCodes_DLL.ini"
```

All response codes and key strings were constants, not magic strings scattered through the logic.

**Step 1 - Parse strIn:**

```vb
Dim strInArraySeperator As String() = New String() {UNIVERSAL_DELIMITER}
Dim strInArray As String() = strIn.Trim().Split(strInArraySeperator, StringSplitOptions.None)
Dim accountNo       As String = strInArray(0).Trim()
Dim transactionCode As String = strInArray(1).Trim()
Dim branchCode      As String = strInArray(2).Trim()
Dim productCode     As String = strInArray(3).Trim()
Dim holderNo        As String = strInArray(4).Trim()
```

`StringSplitOptions.None` was used rather than `RemoveEmptyEntries` - blank fields had to produce empty strings at the correct array positions, not be dropped silently.

**Step 2 - TCodes_DLL.ini check:**

```vb
If ((From line In File.ReadAllLines(TCODES_FILE)
     Where (line.Trim() = transactionCode)
     Select line).Count() > 0) Then
    Return PASS_NEWACCOUNT
    Exit Function
End If
```

A LINQ query against the lines of the INI file. If the transaction code matched any line (trimmed), the account was a new account opening. Return `1000` immediately - no Oracle call.

**Step 3 - Oracle call:**

```vb
Dim con As ADODB.Connection = New Connection()
Dim cmd As ADODB.Command    = New Command()

con.ConnectionString = conStr
cmd.CommandText      = PROCEDURE_NAME
cmd.CommandType      = CommandTypeEnum.adCmdStoredProc

cmd.Parameters.Append(cmd.CreateParameter("branch_cd",   DataTypeEnum.adVarChar, ParameterDirectionEnum.adParamInput,  200, branchCode))
cmd.Parameters.Append(cmd.CreateParameter("product_cd",  DataTypeEnum.adVarChar, ParameterDirectionEnum.adParamInput,  200, productCode))
cmd.Parameters.Append(cmd.CreateParameter("account_no",  DataTypeEnum.adVarChar, ParameterDirectionEnum.adParamInput,  200, accountNo))
cmd.Parameters.Append(cmd.CreateParameter("holder_no",   DataTypeEnum.adVarChar, ParameterDirectionEnum.adParamInput,  200, holderNo))
cmd.Parameters.Append(cmd.CreateParameter("output_txt1", DataTypeEnum.adVarChar, ParameterDirectionEnum.adParamOutput, 3000))
cmd.Parameters.Append(cmd.CreateParameter("output_txt2", DataTypeEnum.adVarChar, ParameterDirectionEnum.adParamOutput, 3000))
cmd.Parameters.Append(cmd.CreateParameter("err_cd",      DataTypeEnum.adInteger, ParameterDirectionEnum.adParamOutput,  200))
cmd.Parameters.Append(cmd.CreateParameter("err_txt",     DataTypeEnum.adVarChar, ParameterDirectionEnum.adParamOutput,  200))

con.Open()
cmd.ActiveConnection = con
cmd.CommandTimeout   = 60
cmd.Execute()
```

All parameters added before `con.Open()`. Connection opened, then `ActiveConnection` set. Timeout 60 seconds. `Execute()` with no return value - output parameters were read back after execution.

**Step 4 - Read output and close:**

```vb
' Close connection first
con.Close()
con = Nothing

' Then read output parameters
errCd  = cmd.Parameters("err_cd").Value
errTxt = cmd.Parameters("err_txt").Value.ToString()

' Then release command
cmd.ActiveConnection = Nothing
cmd = Nothing
```

**Step 5 - Return result:**

```vb
If (errCd = 0) Then
    Return PASS_VALIDACTNUM
Else
    Return PASS_INVALIDACTNUM + UNIVERSAL_DELIMITER + errTxt
End If
```

On exception:

```vb
Catch ex As Exception
    ' Close connection if open
    ' ...
    Return FAIL_EXCEPTION + UNIVERSAL_DELIMITER + ex.Message.ToString
End Try
```

The `FAIL_EOFUNCREACHED` constant at the end of the function:

```vb
ValidateSL210PQ = FAIL_EOFUNCREACHED
```

This was a VB.NET idiom for assigning the function return value without `Return`. It would only be reached if execution somehow fell through all the `Try/Catch/Return` branches - which should never happen. It served as a defensive catch-all.

### 1.4 Method 2: SL210PQ - The Populate Extension

`SL210PQ` shared identical structure with `ValidateSL210PQ` up to and including the Oracle call. The difference was in reading the output parameters when `err_cd = 0`:

```vb
If (errCd = 0) Then
    output_txt1 = cmd.Parameters("output_txt1").Value
    output_txt2 = cmd.Parameters("output_txt2").Value

    ' Release command
    cmd.ActiveConnection = Nothing
    cmd = Nothing

    If flag_populate = True Then
        Return PASS_VALIDACTNUM + UNIVERSAL_DELIMITER + output_txt1 + UNIVERSAL_DELIMITER + output_txt2
    Else
        Return PASS_VALIDACTNUM
    End If
Else
    ' Release command
    cmd.ActiveConnection = Nothing
    cmd = Nothing

    Return FAIL_INVALIDACTNUM + UNIVERSAL_DELIMITER + errTxt
End If
```

Note: `output_txt1` and `output_txt2` were declared at the top of the method, not inside the `Try` block:

```vb
Dim output_txt1 As String = ""
Dim output_txt2 As String = ""
```

This ensured they were always in scope for the return statement regardless of execution path.

The full response when `flag_populate = True` and account is valid:

```
1001 |%|% <customer name line 1> |%|% <customer name line 2>
```

### 1.5 Method 3: ValidateValidTill - Full Source Walkthrough

```vb
<WebMethod()>
Public Function ValidateValidTill(ByVal strIn As String) As String
```

**Comment at the top of the method:**

```vb
'To Call: <Valid Till>|%|%<Document Type>|%|%<Business Type>
```

This was the only inline usage documentation in the entire service. It was sufficient - the format was simple and the delimiter was consistent with the other methods.

**Constants:**

```vb
Const PASS_VALIDENTRY   As String = "1000"
Const FAIL_INVALIDENTRY As String = "9999"
Const UNIVERSAL_DELIMITER As String = "|%|%"
```

Note: `ValidateValidTill` used `9999` for failure, not `1002` or `1003`. This was a different response contract from the account validation methods. The IndexPlus calling code had to handle this correctly.

**Step 1 - Parse and validate input structure:**

```vb
Dim strInArray As String() = strIn.Trim().Split(strInArraySeperator, StringSplitOptions.None)

If UBound(strInArray) <> 2 Then
    Throw New Exception("WS Exception: UBound(strInArray) <> 2; The input(strIn):" + strIn + 
        ", doesn't confirm to the pattern: <Valid Till>|%|%<Document Type>|%|%<Business Type>.")
End If
```

`UBound(strInArray) <> 2` - if splitting on `|%|%` did not produce exactly three elements (indices 0, 1, 2), the input was malformed. The exception message included the actual `strIn` value received - useful for debugging misconfigured IndexPlus setups.

**Step 2 - Extract fields:**

```vb
Dim strData_ValidTill As String = strInArray(0).Trim
Dim document_type     As String = strInArray(1).Trim
Dim business_type     As String = strInArray(2).Trim
```

**Step 3 - Guard against empty document_type or business_type:**

```vb
If String.IsNullOrEmpty(document_type) Then
    Throw New Exception("WS Exception: document_type ... has IsNullOrEmpty true.")
ElseIf String.IsNullOrEmpty(business_type) Then
    Throw New Exception("WS Exception: business_type ... has IsNullOrEmpty true.")
```

A blank `document_type` or `business_type` was a configuration error - a picklist field that should have been pre-validated in IndexPlus before this method was called.

**Step 4 - Branch on Valid Till value:**

```vb
Select Case LCase(strData_ValidTill)
    Case Nothing, ""
        ' Valid Till is empty - check if it is mandatory for this document/business type combination
    Case Else
        ' Valid Till is provided - validate the date
End Select
```

**Empty Valid Till path - INI file lookup:**

```vb
Select Case LCase(business_type)
    Case "ban"  : DocTypes_INI_Files_path = "C:\CaptivaCustom\P_V\DocTypes_INI_Files\BAN_Codes.ini"
    Case "home" : DocTypes_INI_Files_path = "C:\CaptivaCustom\P_V\DocTypes_INI_Files\HOME_Codes.ini"
    Case "cc"   : DocTypes_INI_Files_path = "C:\CaptivaCustom\P_V\DocTypes_INI_Files\CC_Codes.ini"
    Case "mort" : DocTypes_INI_Files_path = "C:\CaptivaCustom\P_V\DocTypes_INI_Files\MORT_Codes.ini"
    Case "nr"   : DocTypes_INI_Files_path = "C:\CaptivaCustom\P_V\DocTypes_INI_Files\NR_Codes.ini"
    Case Else
        Throw New Exception("The Business Type is not either BAN/HOME/CC/MORT/NR. Contact System Administrator.")
End Select

If ((From line In File.ReadAllLines(DocTypes_INI_Files_path)
     Where (line.Trim() = document_type)
     Select line).Count() > 0) Then
    Throw New Exception("You can't enter an empty Date for combination; Document Type: " 
        + document_type + ", and Business Type: " + business_type 
        + ". Please Enter a valid date in format(dd/mm/yyyy).")
Else
    Return PASS_VALIDENTRY
End If
```

If the `document_type` was found in the INI file → date is mandatory → throw exception → return `9999|%|%<message>`.
If not found → date is optional → return `1000`.

**Provided Valid Till path - date validation:**

```vb
Dim enGB As New CultureInfo("en-GB")
Dim dateValue As Date

If Date.TryParseExact(strData_ValidTill, "dd/MM/yyyy", enGB, DateTimeStyles.None, dateValue) Then
    If dateValue > Date.Now Then
        Return PASS_VALIDENTRY
    Else
        Throw New Exception("The date entered:" + strData_ValidTill + 
            ", must be greater than the current date. Please re-enter a valid date in format(dd/mm/yyyy).")
    End If
Else
    Throw New Exception("The date entered:" + strData_ValidTill + 
        ", is invalid. Please enter a valid date in format(dd/mm/yyyy).")
End If
```

`CultureInfo("en-GB")` was important. The date format `dd/MM/yyyy` is ambiguous without specifying culture - `en-US` would interpret `01/12/2026` as January 12, not December 1. `en-GB` was correct for Indian banking dates which followed the British `dd/MM/yyyy` convention.

**Exception handling for the whole method:**

```vb
Catch ex As Exception
    Return FAIL_INVALIDENTRY + UNIVERSAL_DELIMITER + ex.Message.ToString
End Try
Return FAIL_INVALIDENTRY + UNIVERSAL_DELIMITER + "End Of Function Reached."
```

Every exception in `ValidateValidTill` was caught at the outer `Catch` and returned as `9999|%|%<message>`. The operator saw the exception message text directly in the IndexPlus field error display. This meant error messages had to be human-readable - which they were, as seen in the exception strings above.

## Part 2 - Generating the Proxy: wsdl.exe

### 2.1 What wsdl.exe Does

`wsdl.exe` is the .NET SDK Web Services Description Language utility. It reads a WSDL document - the machine-readable contract of a SOAP web service - and generates a strongly-typed client class in the language of your choice. The generated class inherits `SoapHttpClientProtocol` and exposes each web method as a normal .NET method call. No manual SOAP envelope construction. No XML parsing. No HTTP management.

### 2.2 Prerequisites

Before running `wsdl.exe`:

i) Confirm the service is accessible. Browse to `http://[SRV-MASKED]:81/IACustomWebServices/Service1.asmx` in Internet Explorer. The ASP.NET service description page should load.

ii) Confirm the WSDL is accessible. Browse to `http://[SRV-MASKED]:81/IACustomWebServices/Service1.asmx?WSDL`. An XML document should load - this is the contract that `wsdl.exe` will read.

iii) If the development machine routes traffic through an HTTP proxy, add the server IP to IE's proxy bypass list. IE proxy settings affect `wsdl.exe`. See Section 2.4 for the error this causes.

iv) Windows SDK must be installed. `wsdl.exe` is at:
```
C:\Program Files\Microsoft SDKs\Windows\v6.0A\bin\wsdl.exe
```

### 2.3 The Command

```bat
"C:\Program Files\Microsoft SDKs\Windows\v6.0A\bin"\wsdl /language:vb http://[SRV-MASKED]:81/IACustomWebServices/Service1.asmx?WSDL
```

Parameters:

| Parameter | Meaning |
|---|---|
| `/language:vb` | Generate VB.NET. Use `/language:cs` for C#. |
| The URL | The WSDL endpoint of the live service. Must be accessible from the machine running `wsdl.exe`. |

Successful output:

```
Microsoft (R) Web Services Description Language Utility
[Microsoft (R) .NET Framework, Version 2.0.50727.3038]
Copyright (C) Microsoft Corporation. All rights reserved.
Writing file 'C:\Service1.vb'.
```

The generated file is written to `C:\Service1.vb`. Copy it into the Visual Studio 2008 project.

### 2.4 Error: HTTP 407 Proxy Authentication Required

If the machine has an IE proxy configured, `wsdl.exe` may fail:

```
Error: There was an error processing
  'http://[SRV-MASKED]:81/IACustomWebServices/Service1.asmx?WSDL'.
  - The request failed with HTTP status 407: Proxy Authentication Required.
```

Resolution:

i) Open Internet Explorer.

ii) Tools → Internet Options → Connections → LAN Settings → Advanced.

iii) Add the server IP to the Exceptions list.

iv) Click OK. Re-run `wsdl.exe`.

## Part 3 - The Generated Proxy: What wsdl.exe Produced

### 3.1 The Class Declaration

```vb
<System.CodeDom.Compiler.GeneratedCodeAttribute("wsdl", "2.0.50727.3038"),
 System.Diagnostics.DebuggerStepThroughAttribute(),
 System.ComponentModel.DesignerCategoryAttribute("code"),
 System.Web.Services.WebServiceBindingAttribute(Name:="Service1Soap", [Namespace]:="http://tempuri.org/")>
Partial Public Class Service1
    Inherits System.Web.Services.Protocols.SoapHttpClientProtocol
```

Key points:

- `DebuggerStepThroughAttribute` - the debugger will not step into the generated proxy code. It steps through it as a black box. This is correct behaviour - you debug your code, not the generated infrastructure.
- `Partial` - the class can be extended in a separate file without modifying the generated code.
- `SoapHttpClientProtocol` - the base class that handles all SOAP transport mechanics.

### 3.2 The Constructor

```vb
Public Sub New()
    MyBase.New
    Me.Url = "http://[SRV-MASKED]:81/IACustomWebServices/Service1.asmx"
End Sub
```

The endpoint URL was hardcoded into the constructor by `wsdl.exe` at generation time - it read the URL from the WSDL it was pointed at and embedded it. In the IndexPlus DLLs, this was used as-is. No configuration file, no dynamic URL resolution. The service endpoint was fixed for the life of the deployment.

> **Note:** If the service moved to a different server or port, `Service1.vb` had to be regenerated and the project rebuilt. For the AFU deployment this was not a problem - the server did not move during the project lifecycle.

### 3.3 Three Calling Patterns Per Method

`wsdl.exe` generated three variants for each web method. Using `SL210PQ` as the example:

**Pattern 1 - Synchronous (the one used in IndexPlus):**

```vb
<System.Web.Services.Protocols.SoapDocumentMethodAttribute(
    "http://tempuri.org/SL210PQ",
    RequestNamespace:="http://tempuri.org/",
    ResponseNamespace:="http://tempuri.org/",
    Use:=System.Web.Services.Description.SoapBindingUse.Literal,
    ParameterStyle:=System.Web.Services.Protocols.SoapParameterStyle.Wrapped)>
Public Function SL210PQ(ByVal strIn As String, ByVal flag_populate As Boolean) As String
    Dim results() As Object = Me.Invoke("SL210PQ", New Object() {strIn, flag_populate})
    Return CType(results(0), String)
End Function
```

`Me.Invoke` serialises the parameters, constructs the SOAP envelope, sends the HTTP POST, receives the response, deserialises the XML, and returns the result array. From the calling code's perspective: send two parameters, get a string back.

**Pattern 2 - IAsyncResult (Begin/End):**

```vb
Public Function BeginSL210PQ(ByVal strIn As String, ByVal flag_populate As Boolean,
    ByVal callback As System.AsyncCallback, ByVal asyncState As Object) As System.IAsyncResult
    Return Me.BeginInvoke("SL210PQ", New Object() {strIn, flag_populate}, callback, asyncState)
End Function

Public Function EndSL210PQ(ByVal asyncResult As System.IAsyncResult) As String
    Dim results() As Object = Me.EndInvoke(asyncResult)
    Return CType(results(0), String)
End Function
```

The .NET 1.x/2.x async pattern. Begin fires the call; End retrieves the result. Not used in IndexPlus - the Validate() event expected a synchronous result before returning.

**Pattern 3 - Event-based async:**

```vb
Public Event SL210PQCompleted As SL210PQCompletedEventHandler

Public Overloads Sub SL210PQAsync(ByVal strIn As String, ByVal flag_populate As Boolean)
    Me.SL210PQAsync(strIn, flag_populate, Nothing)
End Sub

Public Overloads Sub SL210PQAsync(ByVal strIn As String, ByVal flag_populate As Boolean,
    ByVal userState As Object)
    If (Me.SL210PQOperationCompleted Is Nothing) Then
        Me.SL210PQOperationCompleted = AddressOf Me.OnSL210PQOperationCompleted
    End If
    Me.InvokeAsync("SL210PQ", New Object() {strIn, flag_populate},
        Me.SL210PQOperationCompleted, userState)
End Sub
```

The component model async pattern - fire the call, handle the result in a `Completed` event. Also not used in IndexPlus.

### 3.4 The CompletedEventArgs Classes

For each method, `wsdl.exe` generated a typed `EventArgs` class for the event-based async pattern:

```vb
Partial Public Class SL210PQCompletedEventArgs
    Inherits System.ComponentModel.AsyncCompletedEventArgs

    Private results() As Object

    Friend Sub New(ByVal results() As Object, ByVal exception As System.Exception,
        ByVal cancelled As Boolean, ByVal userState As Object)
        MyBase.New(exception, cancelled, userState)
        Me.results = results
    End Sub

    Public ReadOnly Property Result() As String
        Get
            Me.RaiseExceptionIfNecessary
            Return CType(Me.results(0), String)
        End Get
    End Property
End Class
```

`RaiseExceptionIfNecessary` - if the async call encountered an exception, accessing `.Result` would re-throw it on the calling thread. Clean error propagation in the async model. Not relevant for the synchronous usage in IndexPlus, but worth understanding: the generated class was complete and correct for all three usage patterns.

## Part 4 - Consuming the Proxy from IndexPlus

### 4.1 Adding the Proxy to the Project

After generating `Service1.vb`:

i) In Visual Studio 2008, right-click the project (`AO_IndexPlus` or `TP_IndexPlus`).

ii) Add → Existing Item → select `C:\Service1.vb`.

iii) Confirm `System.Web.Services` is in the project references. If not: right-click References → Add Reference → .NET tab → `System.Web.Services`.

iv) Build. No errors expected from `Service1.vb`.

The project now had two files:

- `Service1.vb` - the generated proxy
- `Source.vb` - the IndexPlus field event classes

### 4.2 How the Proxy Was Called

Instantiation and call in a single expression inside the `Validate()` method:

```vb
Dim wsProxyObject = New Service1
wsResult_Original = wsProxyObject.SL210PQ(strIn, True)
```

`wsProxyObject` was a local variable - a new `Service1` instance per validation call. There was no connection pooling at the proxy level. The proxy object was lightweight; the actual HTTP connection was managed by `SoapHttpClientProtocol` internally.

### 4.3 Parsing the Response

```vb
Dim univDelmtrArr As String() = New String() {UNIV_DELMTR}
Dim wsResult_Code    As String = wsResult_Original.Split(univDelmtrArr, StringSplitOptions.None)(0)
Dim wsResult_Message As String
Dim OUTPUT_TXT1      As String = ""
Dim OUTPUT_TXT2      As String = ""

Select Case wsResult_Code
    Case PASS_NEWACCOUNT    ' "1000"
        info.CurrentNode.Values(wfStep_Custom).SetString("CL0_eICVPrOTxt2", "")
        info.CurrentNode.Values(wfStep_Custom).SetString("CL0_eICVPrOTxt",  "")
        nodeFlagDictionary(info.CurrentNode.Id) = True
        valResult.ValidationPassed = True
        Return valResult

    Case PASS_VALIDACTNUM   ' "1001"
        OUTPUT_TXT1 = wsResult_Original.Split(univDelmtrArr, StringSplitOptions.None)(1)
        OUTPUT_TXT2 = wsResult_Original.Split(univDelmtrArr, StringSplitOptions.None)(2)
        info.CurrentNode.Values(wfStep_Custom).SetString("CL0_eICVPrOTxt2", OUTPUT_TXT2)
        info.CurrentNode.Values(wfStep_Custom).SetString("CL0_eICVPrOTxt",  OUTPUT_TXT1)
        nodeFlagDictionary(info.CurrentNode.Id) = True
        valResult.ValidationPassed = True
        Return valResult

    Case FAIL_INVALIDACTNUM ' "1002"
        wsResult_Message = wsResult_Original.Split(univDelmtrArr, StringSplitOptions.None)(1)
        valResult.ValidationPassed = False
        valResult.Message = wsResult_Message
        Return valResult

    Case FAIL_EXCEPTION     ' "1003"
        wsResult_Message = wsResult_Original.Split(univDelmtrArr, StringSplitOptions.None)(1)
        valResult.ValidationPassed = False
        valResult.Message = wsResult_Message
        Return valResult

    Case Else
        Throw New Exception("Business Logic Error.")
End Select
```

`wfStep_Custom` was `currentBatch.WorkflowSteps.ElementAt(24)` - Step 24 (the Custom step) used as an IA value storage namespace. `SetString` wrote the customer name fields to the batch tree node at page level. These values were picked up by the Documentum Export step.

### 4.4 The nodeFlagDictionary

The cache that prevented repeated service calls for the same node:

```vb
Private Shared nodeFlagDictionary As New Dictionary(Of Integer, Boolean)
```

`Private Shared` - one dictionary instance per DLL lifetime, shared across all `Validate()` invocations for the same Captiva session.

Check at the start of `Validate()`:

```vb
If nodeFlagDictionary.ContainsKey(info.CurrentNode.Id) Then
    If nodeFlagDictionary(info.CurrentNode.Id) = True Then
        valResult.ValidationPassed = True
        Return valResult
    End If
End If
```

Reset in `Changed()` (operator modifies the field value):

```vb
Public Sub Changed(ByVal info As IFieldEventInformation) Implements IFieldEvents.Changed
    If info.CurrentNode.Level = 0 Then
        If nodeFlagDictionary.ContainsKey(info.CurrentNode.Id) Then
            nodeFlagDictionary.Remove(info.CurrentNode.Id)
            nodeFlagDictionary.Add(info.CurrentNode.Id, False)
        Else
            nodeFlagDictionary.Add(info.CurrentNode.Id, False)
        End If
    End If
End Sub
```

Set to `True` after a successful service call. Set to `False` whenever the field value changed. This was the only state management in the IndexPlus DLL - everything else was stateless.

## Summary

The complete SOAP integration comprised four pieces:

| Piece | What It Was | Where It Lived |
|---|---|---|
| `Service1.asmx` | The ASP.NET web service - received SOAP calls, called Oracle, returned results | IIS on `[SRV-MASKED]:81` |
| `web.config` | Oracle connection string - `CustomerDatabase` key | Same IIS application |
| `Service1.vb` (proxy) | Auto-generated client class from `wsdl.exe` - compiled into IndexPlus DLLs | `AO_IndexPlus` and `TP_IndexPlus` VS2008 projects |
| `Source.vb` | IndexPlus field event classes - called the proxy, parsed the response, wrote to IA values | Same VS2008 projects |

The service was written once and consumed by two separate Captiva IndexPlus modules. The proxy generation step - `wsdl.exe` pointed at the live WSDL - meant the client code was always in sync with the service contract. Any change to the service required a proxy regeneration and a DLL rebuild, which surfaced breaking changes at compile time rather than at runtime.
